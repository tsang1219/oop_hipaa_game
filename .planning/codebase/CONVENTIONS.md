## 1. Code Style

### Indentation
- **2 spaces** throughout (TypeScript, TSX, JSON, config files).
- No tabs anywhere in the codebase.

### Quotes
- **Single quotes** for all TypeScript/TSX string literals and imports.
- **Double quotes** only in JSX attribute values (class names, data-testid) and JSON.
- Template literals used for string interpolation and multi-line strings.

```ts
// Typical import
import { useState, useEffect, useCallback } from 'react';

// JSX attributes use double quotes
<div className="text-center" data-testid="button-room-reception">
```

### Semicolons
- **Semicolons present** in most project-authored code (pages, scenes, components, shared schema).
- **No semicolons** in some shadcn/ui-derived files (`button.tsx`, `use-toast.ts`, `utils.ts`). These files were generated by shadcn tooling and kept as-is.
- Rule of thumb: new project code uses semicolons.

### Trailing commas
- Used in multi-line arrays, objects, and function parameter lists.
- Used on the last property in interfaces and object literals.

```ts
const TOWERS = {
  MFA: {
    name: 'MFA Shield',
    cost: 80,
    strongAgainst: ['CREDENTIAL', 'BRUTEFORCE'],  // trailing comma
  },
};
```

### Line length
- No hard limit enforced. Lines commonly reach 120-140 characters, especially in JSX with Tailwind classes.
- Long Tailwind class strings are kept inline, not broken across lines.

### No linter or formatter config
- No `.eslintrc`, `.prettierrc`, or formatting config exists at the project root.
- Style is enforced by convention, not tooling.

## 3. React Patterns

### Component Structure
- **Function components only** -- no class components anywhere.
- `export default function ComponentName()` for pages and most components.
- Named exports (`export function`) for breach-defense modals and PhaserGame (which uses `forwardRef`).

```tsx
// Default export pattern (pages, most components)
export default function BreachDefensePage() { ... }

// Named export pattern (modals, forwarded-ref components)
export function TutorialModal({ title, description, onAcknowledge }: TutorialModalProps) { ... }
export const PhaserGame = forwardRef<Phaser.Game | null, PhaserGameProps>(...);
```

### Hooks Usage
- `useState` for all component-level state (no Redux, no Zustand, no Context for game state).
- `useEffect` for side effects: EventBridge subscriptions, localStorage persistence, scene lifecycle.
- `useCallback` for handler functions passed as props or used in effect dependencies.
- `useRef` for Phaser game instances and mutable values that survive re-renders.
- `useLocation` (from wouter) for navigation.
- `useToast` (custom hook) for toast notifications.
- No `useMemo` observed; computation is done inline.

### State Management
- **No global state store.** All state lives in page-level components (`PrivacyQuestPage`, `BreachDefensePage`).
- **localStorage** used extensively for persistence across page reloads:
  ```ts
  const [completedRooms, setCompletedRooms] = useState<string[]>(() => {
    const s = localStorage.getItem('completedRooms');
    return s ? JSON.parse(s) : [];
  });
  ```
- State is synced to localStorage via individual `useEffect` hooks per state variable.
- Sets are serialized as arrays for localStorage (`JSON.stringify(Array.from(mySet))`).

### EventBridge Pattern (React <-> Phaser)
The canonical pattern for cross-layer communication:
```tsx
// Subscribe in useEffect, clean up on unmount
useEffect(() => {
  const onSomeEvent = (data: { ... }) => {
    // update React state
  };
  eventBridge.on(BRIDGE_EVENTS.SOME_EVENT, onSomeEvent);
  return () => {
    eventBridge.off(BRIDGE_EVENTS.SOME_EVENT, onSomeEvent);
  };
}, [dependencies]);

// Emit from React to Phaser
eventBridge.emit(BRIDGE_EVENTS.REACT_DIALOGUE_COMPLETE);
```

### Conditional Rendering
- Early returns for distinct page modes (gameover, win, hub, exploration):
  ```tsx
  if (pageMode === 'gameover' || pageMode === 'win') {
    return <EndScreen ... />;
  }
  if (pageMode === 'hub') {
    return <HallwayHub ... />;
  }
  // default: exploration view
  ```
- Inline conditionals with `&&` for overlays and modals:
  ```tsx
  {showRecap && recapConcept && (
    <RecapModal concept={recapConcept} onContinue={handleRecapContinue} />
  )}
  ```

### Props Pattern
- Interfaces defined above the component in the same file.
- Destructured in function signature.
- Optional callbacks use `?.:` optional chaining: `onPrivacyScoreChange?.(newScore)`.
- `data-testid` attributes on interactive/display elements for potential testing.

## 4. Phaser Patterns

### Scene Lifecycle

All scenes follow this structure:

```ts
export class SomeScene extends Phaser.Scene {
  // Private class fields with `!` definite assignment
  private player!: Phaser.GameObjects.Sprite;
  private walls!: Phaser.Physics.Arcade.StaticGroup;

  constructor() {
    super({ key: 'SceneName' });
  }

  init(data: { ... }) {
    // Reset all mutable state for scene re-entry
    this.enemies = [];
    this.paused = false;
  }

  preload() {
    // Only in BootScene; other scenes rely on BootScene preloading
  }

  create() {
    // 1. Draw floor/grid
    // 2. Create obstacles/walls
    // 3. Create interactables (NPCs, zones, items)
    // 4. Create player
    // 5. Set up input (cursors + WASD + interact keys)
    // 6. Set up EventBridge listeners
    // 7. Emit SCENE_READY
    eventBridge.emit(BRIDGE_EVENTS.SCENE_READY, 'SceneName');
  }

  update(time: number, delta: number) {
    // Game loop logic; early return if paused
    if (this.paused) return;
  }

  shutdown() {
    // Clean up EventBridge listeners
    eventBridge.off(BRIDGE_EVENTS.SOME_EVENT, this.handler, this);
  }
}
```

### Texture Generation (Programmatic Sprites)
- PrivacyQuest uses **programmatic pixel art** via `Phaser.GameObjects.Graphics`:
  ```ts
  const g = scene.add.graphics();
  g.fillStyle(0x4a90e2);
  g.fillRect(10, 14, 12, 10);
  // ... more pixel drawing
  g.generateTexture('texture_key', 32, 32);
  g.destroy();
  ```
- Generated once in `BootScene.create()` via `generateAllTextures(this)`.
- Idempotent: checks `scene.textures.exists(key)` before regenerating.

### Texture Keys
- Player directions: `player_down`, `player_up`, `player_left`, `player_right`
- NPCs: `npc_receptionist`, `npc_doctor`, `npc_nurse`, etc.
- Objects: `obj_poster`, `obj_manual`, `obj_computer`, `obj_whiteboard`
- Furniture: `furn_desk`, `furn_bed`, `furn_cabinet`, etc.
- BreachDefense sprites: `tower_MFA`, `tower_PATCH`, `threat_PHISHING`, etc. (loaded from PNG)

### Mapping Functions
- `SpriteFactory.ts` exports mapping functions to resolve data IDs to texture keys:
  ```ts
  export function npcTextureKey(npcId: string): string { ... }
  export function furnitureTextureKey(obstacleType?: string): string { ... }
  export function objectTextureKey(spriteType: string): string { ... }
  ```

### Physics
- Arcade physics with zero gravity (`gravity: { x: 0, y: 0 }`).
- Static groups for walls/obstacles.
- Player body: `setSize(24, 24)`, `setOffset(4, 4)`, `setCollideWorldBounds(true)`.
- Diagonal movement normalized: `body.velocity.normalize().scale(MOVE_SPEED)`.

### Input
- Dual input: `cursors` (arrow keys) + `wasd` (WASD keys).
- Named key bindings: `SPACE` for interact, `ESC` for exit.
- Click-to-move via BFS pathfinding in ExplorationScene.
- `Phaser.Input.Keyboard.JustDown(key)` for single-press actions.

### Event Handling
- EventBridge listeners bound with context: `eventBridge.on(EVENT, handler, this)`.
- Cleanup in `shutdown()`: `eventBridge.off(EVENT, handler, this)`.
- Arrow function pattern for handlers that need stable `this`:
  ```ts
  private onDialogueComplete = () => {
    this.paused = false;
  };
  ```

## 5. Import Organization

Imports follow a consistent (but informal) order:

1. **React/framework** (`react`, `react-dom`, `wouter`)
2. **External libraries** (`phaser`, `lucide-react`, `zod`, `@tanstack/react-query`)
3. **Internal Phaser** (`../phaser/PhaserGame`, `../phaser/EventBridge`)
4. **Game data/constants** (`../game/breach-defense/constants`)
5. **Components** (`@/components/...`, `./ComponentName`)
6. **Types** (`import type { ... } from '@shared/schema'`)
7. **Assets** (`import img from '@assets/...'`)

```tsx
// Example from BreachDefensePage.tsx
import { useState, useEffect, useCallback, useRef } from 'react';
import { useLocation } from 'wouter';
import { PhaserGame } from '../phaser/PhaserGame';
import { eventBridge, BRIDGE_EVENTS } from '../phaser/EventBridge';
import { TOWERS, WAVES } from '../game/breach-defense/constants';
import { TUTORIAL_CONTENT } from '../game/breach-defense/tutorialContent';
import { TutorialModal } from '../components/breach-defense/TutorialModal';

import { CodexModal } from '../components/breach-defense/CodexModal';

```

### Path Aliases
- `@/` resolves to `client/src/` (defined in `tsconfig.json` and `vite.config.ts`)
- `@shared/` resolves to `shared/`
- `@assets/` resolves to `attached_assets/`
- Relative paths (`../phaser/...`) used within closely related files.

## 6. Error Handling Patterns

### Minimal error handling
- No try/catch blocks in game code. Errors are not caught in components or scenes.
- Toast notifications for missing scenes:
  ```tsx
  if (!sceneExists) {
    toast({ title: 'Scene Not Found', description: `"${data.sceneId}" is not available yet.`, variant: 'destructive' });
    eventBridge.emit(BRIDGE_EVENTS.REACT_DIALOGUE_COMPLETE);
    return;
  }
  ```
- Server-side: Express error middleware catches unhandled errors and returns JSON:
  ```ts
  app.use((err: any, _req: Request, res: Response, _next: NextFunction) => {
    const status = err.status || err.statusCode || 500;
    const message = err.message || "Internal Server Error";
    res.status(status).json({ message });
    throw err;
  });
  ```

### Defensive coding
- Optional chaining on keyboard input: `this.input.keyboard!.createCursorKeys()` (non-null assertion).
- Nullish defaults in state initializers: `|| 150`, `|| 100`, `|| 'npc_staff'`.
- `Math.max(0, ...)` and `Math.min(100, ...)` for clamping scores/percentages.
- Fallback textures via mapping functions: `return map[key] || 'furn_desk'`.

## 7. CSS / Styling Approach

### Tailwind CSS (primary)
- Tailwind 3 classes used everywhere for layout, spacing, colors, responsive design.
- Extensive use of arbitrary values: `bg-[#1a1a2e]`, `text-[8px]`, `border-[#FF6B9D]`, `shadow-[8px_8px_0px_0px_rgba(0,0,0,1)]`.
- Responsive classes used sparingly: `md:` prefix for some battle screen layouts.

### Design System Colors
- **Dark background**: `#1a1a2e` (used across all pages as base)
- **Accent pink**: `#FF6B9D` (borders, highlights, branding)
- **Success green**: `#2ECC71` / `#27AE60` (buttons, cleared states)
- **Game UI panel**: `#2a2a3e` (HUD backgrounds)
- Tailwind semantic tokens: `text-primary`, `bg-card`, `text-muted-foreground` (used in PrivacyQuest components)
- Direct hex in BreachDefense components (hardcoded pixel-art aesthetic)

### Font
- Pixel font via inline style: `fontFamily: '"Press Start 2P"'` or `font-['Press_Start_2P']`.
- Tiny font sizes: `text-[7px]`, `text-[8px]`, `text-[9px]`, `text-[10px]`.

### Pixel Art Aesthetic
- `imageRendering: 'pixelated'` on canvas containers and sprite images.
- Phaser config: `pixelArt: true`, `roundPixels: true`, `antialias: false`.
- CRT scanline overlay via CSS gradient on canvas wrapper.
- `shadow-[8px_8px_0px_0px_rgba(0,0,0,1)]` for retro drop shadows.
- `border-4 border-black` for thick retro borders.

### Inline Styles
- Used when Tailwind cannot express the value: `background: 'radial-gradient(...)'`, `transform: 'perspective(...)'`.
- Grid layouts use inline `style` for `gridTemplateAreas`.
- Dynamic widths: `style={{ width: `${percentage}%` }}`.

### shadcn/ui
- Base components: `Button`, `Card`, `Toast`, `Tooltip`, `Toaster`.
- `cn()` utility from `clsx` + `tailwind-merge` for conditional class merging.
- `class-variance-authority` (cva) for button variants.

## 8. Common Code Patterns

### Section Comment Dividers
Large files use ASCII art comment dividers:
```ts
// ── State ────────────────────────────────────────────────────
// ── Persistence ──────────────────────────────────────────────
// ── EventBridge listeners ────────────────────────────────────
// ── Phase 1: Wave spawning ─────────────────────────────────
```

### JSDoc Comments
- Scenes and utility files have `/** */` doc comments on classes and exported functions:
  ```ts
  /**
   * ExplorationScene: Renders a PrivacyQuest room in Phaser canvas.
   * Room data is passed via scene.start('Exploration', { room, completedNPCs, ... })
   */
  ```
- Components do not have JSDoc comments.

### Data-Driven Architecture
- Game content stored in JSON files (`gameData.json`, `roomData.json`) and TypeScript constant objects (`constants.ts`, `tutorialContent.ts`).
- Zod schemas in `shared/schema.ts` define the shape; types are inferred with `z.infer<>`.
- Mapping functions (`npcTextureKey`, `furnitureTextureKey`) translate data IDs to rendering keys.

### State Machine Pattern
- Pages use string-literal union types as explicit state machines:
  ```ts
  type PageState = 'START' | 'TUTORIAL' | 'PLAYING' | 'PAUSED' | 'GAMEOVER' | 'VICTORY';
  type PageMode = 'hub' | 'exploration' | 'dialogue' | 'gameover' | 'win';
  type GamePhase = 'dialogue' | 'choices' | 'feedback';
  ```
- Conditional rendering and logic branching is done against these state values.

### Phaser Game Loop Phases
The `BreachDefenseScene.update()` follows a numbered phase pattern:
```ts
update(time: number, delta: number) {
  // Phase 1: Wave spawning
  // Phase 2: Enemy movement
  // Phase 3: Breach detection
  // Phase 4: Tower targeting & firing
  // Phase 5: Projectile movement
  // Phase 6: Cleanup
  // Phase 7: Broadcast state (throttled)
}
```

### UUID Generation
- `Phaser.Math.RND.uuid()` for entity IDs in game scenes.
- `nanoid` in package.json but not observed in use in current source.

### localStorage Keys
Persistence uses string keys, sometimes with room-scoped suffixes:
```
'completedRooms'
'collectedStories'
'completedNPCs'
'completedZones'
'collectedEducationalItems'
'current-privacy-score'
'final-privacy-score'
'gameStartTime'
'resolvedGates_{roomId}'
'unlockedNpcs_{roomId}'
```

### data-testid Convention
Interactive and display elements include `data-testid` attributes:
```tsx
data-testid="button-room-reception"
data-testid="text-privacy-score"
data-testid="container-battle-dialogue"
data-testid="button-acknowledge-tutorial"
data-testid="codex-threat-phishing"
```
Pattern: `{element-type}-{descriptive-name}` in kebab-case.
